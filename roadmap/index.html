v1 Status (What’s actually implemented)

This reflects the last few days of work: the v1 contracts are deployed on Monad testnet and the pipeline is locked with a manifest-first workflow.

Contracts (v1 core)
  • MMMToken (tax logic + wiring setters)
  • TaxVault (custody, one-time RewardVault wiring, only RewardVault can sweep)
  • RewardVault (hold-time + cooldown + minBalance claim gating)
  • Excluded supply rule: eligibleSupply = totalSupply − Σ excludedRewardAddresses
Locked deployment pipeline
  • deploy-v1-locked (deterministic constructor, no guessing)
  • writes deployments/<network>/latest.json (manifest)
  • preflight-from-manifest (reads only manifest, not env)
  • assert-manifest (offline + on-chain checks)
  • stamp-manifest (pins bytecode hash + git commit)
Testnet gating confirmed
  • minHoldTimeSec enforced (new entrant cannot claim)
  • minBalance enforced (below threshold cannot claim)
  • cooldown enforced (claim within cooldown reverts)
  • nothing-to-claim gate behaves correctly
  • live tests passing (except cooldown may be “UnknownRevert” depending on RPC)
What changed vs the “old” codebase

The previous tracker/DEX infra experiments (custom router/factory, SnapshotDividendTracker, WMON mocks, liquidity remover, etc.) were moved out of the v1 deployment surface. v1 is now intentionally minimal: fewer moving parts, fewer assumptions, easier to audit and operate.

Mainnet Readiness

Your mainnet blockers are operational, not architectural: you must create the pair and seed liquidity on Uniswap V2. The router is enforced on mainnet by the locked scripts.

Hard requirements before deploying MMMToken on mainnet
  • Hardhat network configured: monadMainnet (chainId 143)
  • DEX verified: Factory + Router code present
  • WMON address known
  • PAIR_ADDR must be created on mainnet (Factory.createPair)
  • Initial liquidity plan (MMM + MON, LP recipient, price, slippage)
  • Operational plan: when taxes are enabled vs when liquidity is seeded
Mainnet safety checklist (recommended)
  • Finalize and freeze parameters (tax bps, hold time, cooldown, min balance)
  • Confirm owner key security (hardware wallet / segregation / backups)
  • Dry-run scripts: deploy-v1-locked-main.js with DRYRUN=1
  • Preflight from manifest on mainnet
  • Stamp manifest (bytecode hashes + commit hash) and commit to git
  • Monitoring plan (RPC health, balances, RewardVault pending growth, TaxVault inflow)
Key point: PAIR_ADDR is not “automatic” on mainnet

On mainnet, you will generate it by calling UniswapV2Factory.createPair(MMM, WMON), then using that returned pair address as PAIR_ADDR in your deployment. Your script is correctly refusing to proceed without real code at PAIR_ADDR.
